# 数据中台功能演进规划（2026-06-03）

## 背景

当前项目已经具备分析配置、分析配置历史、看板配置、看板配置历史、资源权限、邮件调度、图表渲染、看板拖拽布局、基础数据源管理和单表数据集能力。后续演进不建议继续简单堆页面，而应优先补齐数据中台的核心底座能力，让分析和看板建立在更稳定的数据资产之上。

本规划用于记录后续功能方向和优先级，便于迭代时拆分需求。

## 优先建设

### 1. 数据源管理

目标：把当前固定数据源模式升级为可管理的数据源资产。

已落地基础：

- 新增、编辑、禁用、删除数据源连接
- 数据源列表、搜索、排序和分页
- 查看数据源下的表列表
- 查看表字段、字段类型、字段备注

待补能力：

- 支持测试连接
- 支持数据源级权限控制
- 记录数据源连接变更历史
- 支持真实多连接配置和密钥管理

价值：

- 后续分析不再强依赖固定 MySQL 配置
- 数据源变更可以被管理和审计
- 为多数据源分析、数据集和血缘能力打基础

### 2. 数据集 / 语义模型

目标：让业务用户基于“业务数据集”做分析，而不是每次直接面对物理表字段。

已落地基础：

- 基于物理表创建数据集
- 配置字段别名、字段显示隐藏、字段类型修正
- 定义维度，例如日期、地区、渠道、客户类型
- 定义指标，例如销售额、订单数、转化率、客单价
- 数据集列表、搜索、排序、分页和预览
- 分析页支持选择数据集或物理表，配置会保存 `dataSourceMode`、`datasetId`、`datasetName`
- 看板支持展示基于数据集创建的分析，数据集来源跟随分析配置

待补能力：

- 支持数据集权限
- 邮件快照、血缘和影响分析完整围绕数据集建模
- 看板层展示组件的数据集来源标识和影响关系
- 数据集删除前提示被哪些分析使用

价值：

- 统一指标口径
- 降低分析配置门槛
- 减少物理表字段变化对分析和看板的影响

### 3. SQL 分析模式

目标：给高级用户保留更灵活的分析入口。

建议能力：

- 支持编写 SQL 查询
- 支持 SQL 参数化
- 支持 SQL 结果预览
- 支持将 SQL 保存为分析或数据集
- 服务端限制只允许安全的 `select` 查询
- 对表名、字段名、函数和 limit 做安全约束

价值：

- 满足复杂分析场景
- 不阻塞专业用户
- 与拖拽式分析形成互补

### 4. 分析发布 / 审核

目标：基于已有配置历史能力，区分编辑中的草稿和线上可见版本。

建议能力：

- 分析配置支持草稿版和已发布版
- 保存不立即影响线上看板
- 发布时生成稳定版本
- 看板默认引用已发布分析
- 支持回滚到历史版本
- 可选增加发布审核

价值：

- 避免编辑分析时影响看板展示
- 让历史版本能力真正进入业务流程
- 为协作和权限治理打基础

### 5. 看板分享能力

目标：让看板从内部编辑工具变成可交付的数据视图。

建议能力：

- 生成分享链接
- 支持公开、私有、指定角色访问
- 支持只读访问
- 支持 iframe 嵌入
- 支持导出图片或 PDF

价值：

- 扩大看板使用场景
- 方便跨团队传播数据结果
- 为报表交付和外部展示做准备

## 中期建设

### 6. 数据血缘

目标：建立数据资产、分析和看板之间的影响关系。

建议能力：

- 查看某张表被哪些分析使用
- 查看某个字段被哪些分析使用
- 查看某个分析被哪些看板引用
- 删除或修改数据源、数据集、字段前提示影响范围

价值：

- 降低误删和误改风险
- 提升数据资产可维护性
- 支撑后续治理能力

### 7. 指标目录

目标：集中管理业务指标口径。

建议能力：

- 指标名称、指标描述、指标表达式
- 指标负责人
- 指标所属主题域
- 指标被哪些分析使用
- 指标变更历史

价值：

- 统一业务口径
- 减少重复造指标
- 提升分析结果可信度

### 8. 缓存和刷新策略

目标：提升看板和分析查询性能，并让 Redis 具备明确职责。

建议能力：

- 分析结果缓存
- 看板组件缓存
- 手动刷新
- 定时刷新
- 缓存过期时间配置
- 按数据源、分析、看板维度清理缓存

价值：

- 降低数据库压力
- 提升看板打开速度
- 为高频访问场景做准备

### 9. 告警能力

目标：从“看数据”扩展到“数据异常主动提醒”。

建议能力：

- 指标阈值告警
- 环比、同比异常告警
- 邮件通知
- 告警任务列表
- 告警执行记录
- 后续可扩展企业微信、钉钉通知

价值：

- 提升数据中台的主动性
- 与已有邮件调度能力形成闭环
- 支撑运营监控类场景

### 10. 操作审计

目标：记录关键资源的操作行为。

建议能力：

- 记录谁创建、修改、删除分析
- 记录谁创建、修改、删除看板
- 记录谁修改了资源权限
- 记录谁切换了历史版本
- 支持按资源、用户、时间筛选审计日志

价值：

- 提升权限体系可信度
- 方便问题追溯
- 为生产环境使用做准备

## 暂缓建设

### 1. AI 自动生成看板

当前基础数据模型、数据集和指标目录还不完整，过早做 AI 生成看板容易停留在演示层。建议等数据集、指标和血缘能力稳定后再考虑。

### 2. 复杂工作流编排

除非项目后续明确扩展到数据加工、ETL 或任务依赖编排，否则暂时不建议投入。

### 3. 多租户

如果项目当前只服务单组织或单部署环境，多租户会显著增加权限、数据隔离和部署复杂度。建议在明确商业化或多组织场景后再设计。

## 推荐下一步

建议继续围绕“数据源管理 + 数据集 / 语义模型”补齐权限、影响分析和看板/邮件回归验证。这两项是后续 SQL 分析、指标目录、血缘、缓存和看板分享的基础。补齐后，项目会更接近真正的数据中台，而不是单纯的 BI 分析工具。
