# 分析上卷 / 下卷需求记录（2026-06-04）

## 背景要求

当前分析页已经具备 `dimensions = 分组/维度`、`measures = 值/指标` 的字段角色，以及基于维度和值动态生成 SQL 的能力，但还不支持 BI 常见的上卷 / 下卷交互。

目标能力：

- 下卷：从更粗维度进入更细维度。例如 `省份 -> 城市 -> 门店`，点击“浙江”后按城市汇总，再点击“杭州”后按门店汇总。
- 上卷：从当前细维度返回上一层维度。例如从门店返回城市，再返回省份。
- 已下钻路径作为运行态过滤条件，当前层级维度进入 `GROUP BY`，值字段继续按 `measures` 聚合。
- 上卷 / 下卷应基于分析字段语义实现，不只是前端对当前结果做筛选。

## 架构判断

当前架构适合做该功能，原因是：

- `dimensions` 已经表示分组维度，可以天然作为钻取层级来源。
- `measures` 聚合语义已经收口，上卷 / 下卷不需要重写值字段聚合逻辑。
- `AnalyzeQueryBuilder` 已经统一处理 `WHERE / GROUP BY / HAVING / ORDER BY`，运行态钻取路径可以转成额外过滤条件。
- 分析页查询入口集中在 `useAnalyzeDataHandler`，适合先实现分析页运行态 MVP。
- 看板和邮件复用分析配置，后续可以在 MVP 稳定后再扩展。

## 分阶段计划

第一阶段先做分析页 MVP，不入库复杂配置：

1. 使用当前 `dimensions` 顺序作为钻取层级。
2. 默认只用第 1 个维度参与查询和图表 X 轴。
3. 点击图表中的某个维度值后，下卷到下一个维度。
4. 已点击的维度路径追加为运行态过滤条件。
5. 顶部展示钻取面包屑，支持点击任意上级节点上卷。
6. 表格、柱状图、折线图、饼图先按统一规则处理；若表格交互复杂，可先只做图形图表，再补表格。
7. 不改变已保存的正式配置结构，只维护分析页运行态 drill state。

第二阶段再做产品化增强：

1. 增加是否启用钻取的配置。
2. 支持用户指定参与钻取的维度层级，而不是默认使用全部 `dimensions`。
3. 看板 widget 支持独立上卷 / 下卷状态。
4. 看板组件展示当前钻取路径和数据集来源。
5. 邮件快照明确只渲染默认层级，或支持指定快照钻取路径。
6. 将钻取配置保存到 `commonChartConfig` 或独立配置字段，并纳入历史版本。

## 第一阶段改动清单

- 新增分析页钻取状态：当前层级、已下钻路径、是否可上卷 / 下卷。
- 查询参数生成时，根据当前层级只传入当前维度作为 `dimensions`。
- 将已下钻路径转换为运行态 `raw` 过滤条件，并和页面已有筛选合并。
- 图表层暴露点击维度值事件，分析页接收后触发下卷。
- 增加钻取面包屑 UI，用于展示路径和触发上卷。
- 进入新分析、切换数据源、清空维度、切换历史版本时重置钻取状态。
- 补 `pnpm test:analyze-query` 测试，覆盖“路径过滤 + 当前维度分组 + measures 聚合”。

## 风险和约束

- 必须先基于 `dimensions` 命名实现，不再回到旧 `groups` 语义。
- 下卷路径过滤要和用户已有筛选条件合并，避免覆盖用户筛选。
- 对日期维度下卷时仍要复用统一日期表达式，避免展示粒度和 SQL 过滤粒度不一致。
- 图表点击值需要使用原始维度值，不能只取格式化后的展示文本。
- MVP 阶段不建议同时做看板持久化和邮件钻取快照，避免一次改动过大。
